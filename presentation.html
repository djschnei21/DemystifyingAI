The whole system in one sentence

A coding agent is not just a model.

A coding agent is a stateless model API wrapped in a purpose-built harness that manages context, tools, permissions, and feedback loops.

Most of the engineering is outside the model: request construction, tool execution, state management, safety gates, and verification.

Compressed model - request

The request is the atomic unit.

The model does not remember your project. Every turn is a model request assembled by the harness.

Prompt

System rules, role, style, policy, and developer instructions.

Task

The current user request or next agent-loop objective.

Context

Relevant history, file snippets, search results, and tool outputs.

Capabilities

Tool definitions, schemas, and resource descriptions the model can use.

Continuity is constructed. If it matters, the harness has to put it into the current request.

Compressed model - harness

Harnesses are purpose-built runtimes.

A harness is the code around model request(s). Different harnesses are built around different control patterns.

Chat harness

User-driven turns

The user supplies the next prompt. The harness packages context and returns a response.

Workflow harness

Program flow

The path is predefined in software. The model handles bounded steps inside the sequence.

Agent harness

Model-driven loop

The model chooses the next step or tool. The harness executes only what is allowed.

Compressed model - agent

Agent means model-driven control flow.

Workflow

Code chooses the path

The harness follows a predefined sequence: classify, retrieve, draft, evaluate, send.

Agent loop

Model chooses the next step

The model asks to read files, run tests, edit code, inspect errors, and continue until done.

The harness still controls reality. The model can propose actions; the harness validates, authorizes, executes, records, and stops the loop.

Compressed model - tools

Tools are the capability boundary.

Advertise

The harness exposes names, descriptions, and schemas.

Propose

The model emits a tool name and structured arguments.

Gate

The harness validates arguments and checks permissions.

Record

The result becomes context for the next model request.

The model proposes. The harness disposes. Nothing touches files, shell, network, or APIs until the harness executes it.

Compressed model - practical stack

The practical coding-agent stack.

Model

Stateless capability: code understanding, language, planning, tool-call proposals.

Harness

State, context selection, permissions, loop control, tracing, and verification.

Tools / MCP

Discoverable capability surface for filesystem, shell, APIs, docs, and services.

Skills / prompts

Procedure, policy, conventions, examples, and task-specific guidance.

Sub-agents

Scoped child loops for isolation, parallelism, and specialization.

Workspace

The real environment where side effects happen and ground truth is observed.

If time is short, remember this stack. The rest of the deck unpacks each layer.

The whole talk in one slide

There is no magic.

Everything we are about to look at is one of three things:

Model
A stateless service over HTTP
Weights on a GPU behind an API. Pure function: tokens in, token probabilities out.
Harness
Code around request(s)
A purpose-built runtime that builds requests, stores state, exposes tools, gates permissions, and records what happened.

The model is a commodity. The purpose-built harness is the product.

01

The LLM as a hosted service.

Start with the boring part: a model is an HTTP service backed by GPU memory.

The LLM as a hosted service - physical model

Weights on GPUs. HTTP at the edge.

Weights

A production LLM is a set of binary weight files loaded into GPU memory. At inference time those weights are read, not rewritten.

Service

The serving layer accepts HTTP/JSON requests, runs the model, and streams generated tokens back to the caller.

Stateless

The server does not remember your conversation between calls. Any continuity must arrive again as input text.

Cost shape

Longer prompts and longer outputs cost more because the model must process more tokens through the same fixed weights.

The LLM as a hosted service - stateless inference

A forward pass does not remember.

f(weights, input_tokens) → next_token_probabilities Same weights + supplied input tokens → a distribution for what token should come next.

Generation is repeated next-token prediction. The model sees the tokens you send in this request, predicts what comes next, appends that token, and repeats until it stops.

The model only knows the current request. If a fact, file, or prior turn matters, the harness has to include that text in the API call it sends now.
02

The API request,
conceptually.

A JSON POST. That's it. A multi-turn agent conversation is multiple stateless calls, each carrying the relevant history again.

The API request, conceptually - request body

Every turn is a request body.

POST /model-api
{
  "model": "some-model-id",
  "messages": [system, user, prior_turns, tool_results],
  "tools": [read_file_schema, run_bash_schema, apply_edit_schema],
  "max_tokens": 4096,
  "stream": true
}

The exact provider shape varies. The important part is the harness builds one complete model invocation at a time.

The API request, conceptually - what comes back

Two things can come back.

Assistant text

Answer or clarification

Plain text for the user. This may be the final answer, a status update, or a question before doing more work.

Tool-call request

A typed action proposal

Structured arguments naming a tool and inputs. The model proposes the action; the harness decides whether and how to run it.

The response is data. Even a tool call is still just output from the model until the harness validates and executes it.

The API request, conceptually - memory

Memory is rebuilt, not remembered.

The model invocation is independent. Continuity comes from the harness rebuilding the next request. Prior turns, file snippets, tool results, summaries, and retrieved docs only matter if the harness includes them again.

The model does not own the conversation. The request body does. Context engineering is deciding what belongs in that request right now.

The API request, conceptually - context budget

The context window is the budget.

History

Prior user messages, assistant responses, tool outputs, summaries, and decisions compete for space.

Workspace context

Files, diffs, test output, logs, search results, and retrieved docs are only useful if included in the current call.

Tool definitions

The available functions and their schemas also consume prompt budget before the model can call them.

Output room

The model's answer needs space too. If the prompt consumes the whole budget, useful generation gets squeezed.

This is why the harness matters. Bad answers often come from missing, stale, or buried context, not from the model randomly forgetting.

03

Harnesses.

Section 2 showed that the request is the unit of work. The harness is the purpose-built code around one or more model requests.

Harnesses - purpose-built runtime

A harness wraps model request(s).

It can be built for chat, for a fixed workflow, or for an agent loop. The difference is who drives the next step.

request = harness.build_request(user_input, context, tools)
response = model_api.call(request)

# Then the harness follows its interaction model:
chat: return_to_user(response)
workflow: run_next_coded_step(response)
agent: execute_tool_and_loop(response)

Harness is not synonymous with agent. A coding-agent harness is a harness purpose-built for the agent-loop interaction model.

The harness - what the harness owns

All the state. All the side effects.

Request construction & state

  • Prompt assembly — identity, rules, environment, project config
  • Context selection — files, snippets, search results, test output, summaries
  • History management — append, compact, truncate, and replay useful state
  • Telemetry & tracing — record each request, response, and decision

Execution & safety

  • Tool registration — capabilities → JSON Schema
  • Execution boundary — filesystem, shell, network, and APIs
  • Policy boundary — approvals, command blocking, sandboxing, isolation
  • Feedback loop — tests, linters, errors, retries, and verification
  • Delegation — scoped child runs and merged summaries
The harness - why this matters

The harness is the product.

OpenCode, Claude Code, and Copilot can call similar models and still feel different because their coding-agent harnesses make different choices.

OpenCode

Terminal-native coding-agent harness used heavily by the team. Selectable agents/profiles, local tools, permissions, and child runs all sit around the same loop.

Claude Code

Purpose-built coding-agent harness with strong permission flow, skills, sub-agents, hooks, and a polished execution loop.

GitHub Copilot agent mode

IDE-integrated coding-agent harness that brings the loop into editor context, repo state, and familiar VS Code workflows.

Others

Cursor, Aider, Cline, Codex CLI, Goose, Devin, OpenHands: different UX and policy choices around model calls, context, tools, and edits.

The base model matters, but the harness determines the working experience: interaction model, context selection, tool surface, permissions, edit mechanics, and feedback loops.

04

Interaction
models.

Harnesses are usually purpose-built around one primary control pattern.

Interaction models - who drives the next step

Chat. Workflow. Agent loop.

Chat harness

User-driven turns

The user or app drives each turn. The harness builds a request, returns a response, and waits for the next user action.

Workflow harness

Code-driven steps

The harness follows predefined code paths. The model fills in specific steps, but code decides the sequence.

Agent harness

Model-driven loop

The model chooses the next step or tool. The harness executes allowed actions, observes results, and calls the model again.

The distinction is control flow, not intelligence. Chat is user-driven. Workflow is code-driven. Agent loop is model-driven within harness limits.

Interaction models - naming

Different words. Same control split.

User turn

Chat or assistant

The user supplies the next prompt. The harness packages context, calls the model, returns a response, and waits.

Program flow

Workflow, chain, pipeline

The path is predefined by the harness. The model handles bounded steps inside that coded sequence.

Model loop

Agent or agentic system

The model picks the next step or tool. The harness still owns execution, state, permissions, and stopping conditions.

“Agent” is overloaded. In this deck, it means model-driven control flow inside a harness, not magic and not unchecked autonomy.

05

Tools &
function calling.

From here, focus on coding-agent harnesses. Tools can appear in workflows too, but they matter most when the model chooses the next action.

Tools & function calling - the contract

A tool call is a typed request.

Advertise

The harness includes tool names, descriptions, and schemas in the request body it sends to the model.

Propose

The model emits a tool name and structured arguments. It is asking for work, not doing the work.

Gate

The harness validates the arguments, checks permissions, and decides whether the action is allowed.

Record

The result, error, or denial is appended to history and becomes context for the next model call.

The response is still data. In an agent loop, the model discovers and requests capabilities; nothing touches disk, shell, network, or APIs until the harness chooses to execute it.

Tools & function calling - the wire format

Schema in. Request out.

Tool definition in the request

{
  "name": "run_bash",
  "description": "Run a command",
  "parameters": {
    "command": "string",
    "timeout_seconds": "number"
  }
}

Model output

{
  "tool": "run_bash",
  "arguments": {
    "command": "pytest",
    "timeout_seconds": 120
  }
}

The description is part of the prompt. It helps the model decide when a tool is relevant; the schema tells it what shape the arguments should have.

Tools & function calling - validation

The schema helps. The harness still gates.

Shape

The schema nudges the model toward the expected fields and types. It makes tool calls easier to parse.

Validate

The harness rejects malformed arguments, missing required fields, invalid paths, or unsupported options.

Authorize

Dangerous or high-impact calls require approval, sandboxing, or denial before anything executes.

Feedback

Success, failure, and denial are appended as tool results so the model can choose the next step.

Tools & function calling - the recurring tool surface

Most tools are boring.

Read

read_file list glob grep

Bring workspace state into the next request.

Edit

apply_edit write_file

Change files, usually through diffs or patch operations the harness can inspect.

Run

run_bash run_tests run_linter

Execute commands and turn output into feedback.

Delegate / extend

task web_fetch MCP tools

Spawn scoped work or connect to external services.

OpenCode, Claude Code, and Copilot expose different names and UX. In an agent harness, this surface must be discoverable because the model is choosing among read, search, edit, run, verify, delegate, and extend.

06

MCP.

A standard way for agent harnesses to discover and invoke external tools, resources, and prompts.

MCP - what it is

MCP is not a model protocol.

The model never speaks MCP directly. The harness speaks MCP to servers, then translates discovered capabilities into model-visible context and tool definitions.

Host

The user-facing harness or app: editor, CLI, desktop client, or coding agent.

Client

An MCP client embedded in the host. Usually one client connection per server.

Server

A local or remote process exposing tools, resources, prompts, and capabilities.

Model

Sees the translated descriptions in the request. It does not open MCP connections.

MCP standardizes host-to-server discovery and invocation. That discovery matters most when an agent loop needs a model-visible capability surface.

MCP - what it standardizes

Discovery. Invocation. Results.

Tools

Callable actions the server exposes. The harness can present them to the model as tool definitions.

Resources

Read-only context the host can fetch and include in a model request when relevant.

Prompts

Server-supplied templates or workflows that can help shape the next request.

Capabilities

The host and server negotiate what each side supports during initialization.

Same boundary as tools. The model can request an action; the harness decides whether and how to invoke the MCP server.

MCP - wire format

JSON-RPC over stdio or HTTP.

Session

A stateful initialize handshake negotiates protocol version and capabilities between host client and server.

Transports

stdio for local subprocess servers. Streamable HTTP for remote servers.

Messages

Requests, responses, and notifications flow over the connection until it closes.

Model API

Still separate and stateless. The harness folds MCP results into later model requests.

MCP - security boundary

MCP expands the tool surface.

Prompt surface

Tool names, descriptions, annotations, resources, and prompts can enter the model request. Treat server-provided text as untrusted unless the server is trusted.

Execution surface

MCP servers can expose powerful actions. The harness still owns auth, allowlists, permissions, confirmation, and logging.

The model does not execute MCP tools. It emits requests; the host/harness invokes the server if policy allows.

07

Skills.

Skills are context packages for agent harnesses: cheap metadata first, detailed instructions only when relevant.

Skills - what a skill actually is

A folder. A markdown file. That's it.

my-skill/
├── SKILL.md       # required: YAML + markdown
├── scripts/       # optional: Python, Bash, JS
├── references/    # optional: docs, loaded on demand
└── assets/        # optional: templates, schemas

Versionable, reviewable, and shareable as plain files. No framework-specific code required.

---
name: pdf-processing
description: Extract text and tables from PDF
  files, fill forms, merge documents. Use when
  working with PDFs or when the user mentions
  forms or document extraction.
---
# PDF Processing
## Quick start
Use pdfplumber to extract text from PDFs:
```python
import pdfplumber
with pdfplumber.open("doc.pdf") as pdf: ...
```
For form filling, see forms.md.
Skills - the architectural trick

Progressive disclosure.

Skills start as cheap metadata. The harness loads detail only when the task needs it.

1 · Discovery

The harness exposes name + description so the model knows what exists.

name + description

Most bodies stay out of the request.

2 · Loading

When selected, the harness reads SKILL.md and includes it in a later request.

SKILL.md

Detailed guidance is paid for when relevant.

3 · Use

Instructions shape tool calls and verification. Bundled files are resources, not actions.

references + scripts

Running a script still goes through a tool.

Skills - three things, often confused

Tools vs skills vs prompts.

What it isWhen loaded
ToolA callable capability with a JSON-Schema signatureExposed — available to this turn when the harness advertises it
SkillA bundle of instructions, scripts, and referencesLoaded — harness includes the body after model selection / request
System promptBaseline identity, behavior, and policy rulesIncluded — assembled by the harness for the request

Conceptually: tools are verbs. Skills are know-how. Prompts are policies. In agent harnesses, this separation helps the model choose actions without confusing instructions with executable capabilities.

Skills - what changes

What skills change. What they do not.

Add procedure

Skills add conventions, examples, checklists, scripts, references, and domain-specific operating procedures.

Shape tool use

The model can use the loaded instructions to choose tools, sequence steps, and verify work more consistently.

No private memory

A skill does not make the model remember anything outside the current request. The harness still has to include the relevant context.

No bypass

A skill does not execute itself or skip permissions. Even bundled scripts still run through an allowed tool path.

The agent loop does not change. Skills add procedural context and resources; tools perform actions; the harness gates side effects.

08

Sub-agents &
coordination.

Once an agent harness has tools, MCP, and skills, it can compose multiple scoped loops with isolated context.

Sub-agents & coordination - sub-agents

Same loop. Isolated context.

A sub-agent is another instance of the same loop, launched with:

  • A separate, isolated context window
  • A scoped task described by the parent
  • Possibly a reduced tool set & stricter permissions
  • Possibly a different model

Mechanically: the OS spawning a child process. Parent passes initial state. Child runs its own loop. Usually the parent receives a summary or selected result; intermediate noise can stay in the child context.

The important property is isolation: the child can search, run commands, and accumulate noisy context without polluting the parent loop.

Why you'd want one

Context isolation

Running tests can produce 50k tokens of output you don't need in the main window.

Parallelism

Investigate auth, DB, and API modules simultaneously and synthesize.

Specialization

A read-only “code-reviewer”; a sandboxed “test-runner” that can't touch source.

Sub-agents & coordination - the cost reality

Sub-agents aren't free.

Every spawned loop has its own context, requests, tool calls, and summaries. Useful planning anchors:

Chat call

1×tokens

Single request/response. Baseline.

Single-agent loop

~4×tokens

Multi-turn loop with tool calls.

Multi-agent system

~15×tokens

Parent + parallel sub-agents.

Use sub-agents when the gain justifies the cost. Coordination overhead grows non-linearly. Nesting beyond 1–2 levels rarely pays off.

Sub-agents & coordination - common patterns

Common coordination patterns.

Agents as tools

A parent loop keeps control and calls specialist agents for bounded subtasks.

Handoff

Control transfers to a specialist agent for the next part of the interaction.

Orchestrator-workers

A central loop breaks down an open-ended task, fans out work, and synthesizes results.

Evaluator-optimizer

One loop produces an answer; another evaluates it and feeds back improvements.

Names vary by framework. The mechanics are what matter: start scoped loops, control what context they see, and merge back only the useful result.
